# Architecture (current state)

Snapshot of what is scaffolded in the repo today. No forward plans — see
issues / prompts for that.

## Repo layout

Two independent projects, each with its own `package.json`, `tsconfig`,
and `wrangler.toml`. No workspace.

```
frontend/   SvelteKit app, deployed to Cloudflare Pages
backend/    Hono Worker, deployed as a standalone Cloudflare Worker
poc/        Static audio-control POCs (unchanged by scaffold)
```

## Frontend (`frontend/`)

- SvelteKit 2 + Svelte 5 (runes) + TypeScript, adapter-cloudflare.
- Tailwind v4 via `@tailwindcss/vite`, with `@custom-variant dark`.
- shadcn-svelte, vega preset, zinc base color. Config hand-written at
  `components.json` (see Deviations). Starter components:
  `$lib/components/ui/{button,input,label,card}`.
- Paraglide JS (inlang) for i18n. Locales: `en`, `zh`. Messages live in
  `messages/*.json`. Compiled output at `src/lib/paraglide/` is
  `.gitignore`d and regenerated by the Vite plugin on dev/build.
- `wrangler.toml` declares `compatibility_flags = ["nodejs_compat"]`
  (Paraglide's server middleware needs `AsyncLocalStorage`). No DB,
  R2, or KV bindings — frontend owns none.
- `BACKEND_URL` is a plain `[vars]` entry, overrideable via `.dev.vars`
  locally and Pages env in prod. Typed on `App.Platform.env`.

### Thin BFF layer

SvelteKit server routes act as a narrow proxy:

- `src/lib/server/api.ts` exports `createApiClient({ backendUrl,
  sessionToken, fetch })`. Always called from `+server` / `+page.server`
  / `load` — never from the browser — so `BACKEND_URL` and the session
  token never ship to the client bundle.
- Session cookie name (`session`) is pulled from the server request and
  re-attached host-to-host when calling the backend. No CORS in the
  browser path because the browser only ever talks to its own origin.
- Sample: `GET /api/health` at `src/routes/api/health/+server.ts`
  forwards to backend `/health`.

## Backend (`backend/`)

- Hono 4 on Cloudflare Workers. Entrypoint `src/index.ts` exports the
  default app.
- CORS middleware mirrors `FRONTEND_ORIGIN` back with `credentials: true`.
  In prod the frontend talks to the backend host-to-host from server
  routes, so browser CORS mostly doesn't trigger; the middleware is
  there for direct dev calls and future client-side fetches.
- Drizzle ORM wired to D1. `createDb(env.DB)` is installed on every
  request as `c.var.db`. `src/db/schema.ts` is currently empty — domain
  tables (users, sessions, tracks, playlists, playlist_tracks) land per
  feature.
- `drizzle.config.ts` uses dialect `sqlite`, driver `d1-http`.
  Migrations will land in `drizzle/` once the first table ships.
- `wrangler.toml` declares bindings for D1 (`DB`), R2 (`R2`), and KV
  (`KV`) with `<fill-me>` IDs. Created via `wrangler d1 create`,
  `wrangler r2 bucket create`, `wrangler kv namespace create`.
  `compatibility_flags = ["nodejs_compat"]` is set.
- `.dev.vars.example` lists `SESSION_SECRET` + `FRONTEND_ORIGIN`. The
  real `.dev.vars` is git-ignored; prod uses `wrangler secret put`.

### Auth primitives

Library code only; no routes, no DB persistence yet (blocked on schema).

- `src/auth/password.ts` — `hashPassword`, `verifyPassword` via
  `hash-wasm` argon2id (WASM, runs on Workers). Params: m=19 MiB, t=2,
  p=1, 32-byte output.
- `src/auth/session.ts` — `generateSessionToken` (20-byte base32),
  `hashSessionToken` (SHA-256). Oslo-style: client holds the random
  token, DB will store only its hash.

## Request flow

```
browser ──► frontend (Pages, SvelteKit SSR) ──► backend Worker ──► D1 / R2 / KV
           │ session cookie set here                │ reads session cookie from
           │ pages use $lib/server/api.ts to call  │ forwarded `cookie:` header
           │ backend over HTTP                      │
           ▼
       rendered page / JSON response
```

All bindings are on the backend. The frontend only sees HTTP responses
from the backend Worker.

## Bindings ownership

| Binding          | Owner    | Notes                          |
| ---------------- | -------- | ------------------------------ |
| D1 `DB`          | backend  | Drizzle on top                 |
| R2 `R2`          | backend  | audio bucket `r-listener-audio`|
| KV `KV`          | backend  | cache namespace                |
| `BACKEND_URL`    | frontend | plain var, points at Worker    |
| `FRONTEND_ORIGIN`| backend  | CORS allowlist                 |
| `SESSION_SECRET` | backend  | `wrangler secret put` in prod  |

## Deviations from `tech-stack.md`

- **Backend/BFF split.** tech-stack.md originally said the BFF was
  "SvelteKit server routes"; it now means a thin proxy layer only.
  The real backend is a separate Hono Worker. tech-stack.md has been
  updated in parallel.
- **Argon2 implementation.** Originally `@node-rs/argon2`. Swapped to
  `hash-wasm` because `@node-rs/argon2` uses native bindings and does
  not load on the Cloudflare Workers runtime. Same algorithm (argon2id),
  same OWASP-tier parameters.
- **shadcn-svelte init.** The `shadcn-svelte init` CLI gates on an
  interactive preset picker and its `--preset` flag only accepts URL-
  encoded preset codes from shadcn-svelte.com/create, not preset names.
  `components.json` was written by hand (vega / zinc) and individual
  components are added via `shadcn-svelte add <name>`.

## What is not scaffolded

- Any domain tables or migrations.
- Auth routes / session persistence / hooks.
- Real pages beyond the default `+page.svelte`.
- Cloudflare Queues / Cron Triggers.
- GitHub Actions workflows for deploy.
